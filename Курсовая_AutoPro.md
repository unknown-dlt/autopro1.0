## Введение

Современные автосервисы нуждаются в удобных инструментах для учёта записей клиентов, складских остатков и работы сотрудников. Бумажные журналы и таблицы в Excel не позволяют быстро видеть расписание, остатки запчастей и отчёты в одном месте. Поэтому актуальной задачей является разработка веб-приложения, которое объединяет расписание, склад, механиков и отчётность в едином интерфейсе.

**Объект исследования** — процесс учёта и планирования работ в автосервисе.  
**Предмет исследования** — проектирование и реализация веб-приложения для автоматизации этого процесса.

**Цель работы** — разработать веб-приложение AutoPro для учёта записей клиентов, управления складом запчастей, учёта механиков и формирования отчётов.

**Задачи:**

- изучить требования к подобным системам и выбор технологий;
- спроектировать структуру приложения (фронтенд и бэкенд);
- реализовать интерфейс: авторизация, расписание, склад, механики, отчёты, история, уведомления;
- реализовать серверную часть с хранением данных и API;
- проверить работоспособность и удобство использования.

**Методы:** анализ предметной области, проектирование структуры данных и интерфейсов, разработка на JavaScript (Vue.js, Express), тестирование в браузере.

**Практическая значимость:** приложение можно использовать в небольшом автосервисе для ведения расписания, склада и отчётности без покупки коммерческого ПО.

## Глава 1. Теоретические основы и постановка задачи

### 1.1. Анализ предметной области

Автосервис ведёт приём заявок от клиентов, назначает время приёма и закрепляет работы за механиками. Для работ требуются запчасти, которые списываются со склада. Руководству нужны отчёты по выручке, по количеству работ и по зарплате механиков. Таким образом, ключевые сущности системы: клиенты (в контексте записей), записи на обслуживание (дата, время, услуга, механик, статус, детали), склад (позиции с артикулом, названием, ценой, количеством), механики (ФИО, должность, дата приёма, активность), услуги, история завершённых работ и уведомления о событиях.

### 1.2. Выбор технологий

Для реализации выбран стек, позволяющий быстро разработать и развернуть приложение без сложной инфраструктуры:

- **Фронтенд:** Vue.js 3, Vue Router, Pinia, Vite. Vue даёт удобную реактивность и компонентный подход; Vite ускоряет сборку и разработку.
- **Бэкенд:** Node.js, Express. Простой REST API, без базы данных на первом этапе — данные хранятся в JSON-файле (backend/data.json), что достаточно для курсового проекта и небольшого сервиса.
- **Стили:** обычный CSS с общими переменными и анимациями для единого визуального стиля (фиолетовая палитра, карточки, кнопки, модальные окна).

Такой выбор позволил сосредоточиться на логике приложения и интерфейсе, а не на настройке СУБД и сервера.

### 1.3. Постановка задачи

Требовалось реализовать:

1. **Авторизацию** — вход под ролью «Менеджер» или «Механик» с простой капчей.
2. **Главную страницу (Обзор)** — карточки: выставлено счетов (сумма по активным заказам), запчасти на складе (шт. и сумма), количество записей; расписание на сегодня; активные механики; график работ за месяц.
3. **Расписание** — календарь по месяцам, выбор дня, список записей по часам с привязкой ко времени, форма создания/редактирования записи (клиент, телефон, авто, госномер, VIN, услуга, механик, статус, необходимые детали). При завершении записи — списание деталей со склада. Двойной клик по записи — просмотр и редактирование в модальном окне.
4. **Каталог услуг** — добавление, редактирование, удаление услуг (название, описание, длительность, цена).
5. **Склад запчастей** — добавление и удаление позиций (артикул, название, цена, количество); при завершении записи указанные детали списываются.
6. **Механики** — учёт сотрудников (ФИО, должность, дата приёма), переключение активности.
7. **Историю обслуживаний** — фильтр по госномеру, список завершённых работ; двойной клик — детали записи (как в расписании).
8. **Отчёты** — период ГГГГ-ММ, выручка за период, количество завершённых заявок, разбивка по услугам, фонд оплаты труда механиков (оклад + доплата за работу).
9. **Уведомления** — в шапке: новые записи, изменения статусов записей, изменения по механикам и складу.

Дополнительно: ограничения ввода в полях (имя — только буквы, телефон — цифры и формат, год — 4 цифры, госномер — по маске, VIN — 17 символов и т.д.), анимации и единый визуальный стиль.

## Глава 2. Проектирование и реализация системы

### 2.1. Структура проекта

Проект состоит из корневой папки с общим package.json (скрипт `npm run dev` запускает и фронтенд, и бэкенд через concurrently), папки frontend (Vue + Vite) и backend (Express).

**Frontend:**  
- Точка входа — main.js, роутер — router/index.js (маршруты: логин, /app/dashboard, /app/schedule, /app/services, /app/parts, /app/mechanics, /app/history, /app/reports).  
- Store (Pinia) — хранение токена и роли после входа.  
- Layout MainLayout.vue — сайдбар с меню, шапка с уведомлениями и выходом, область контента.  
- Страницы-компоненты: LoginPage, DashboardPage, SchedulePage, ServicesPage, PartsPage, MechanicsPage, HistoryPage, ReportsPage.  
- Общие стили и утилиты: main.css, inputFilters.js (фильтры ввода для полей).

**Backend:**  
- server.js — Express, CORS, парсинг JSON, чтение/запись data.json.  
- Маршруты: /api/login (проверка роли, пароля, капчи), /api/dashboard, /api/services, /api/parts, /api/mechanics, /api/appointments, /api/history, /api/reports, /api/notifications.  
- Для доступа к API (кроме login) требуется заголовок Authorization: Bearer <token>.  
- При смене статуса записи на «Завершён» выполняется списание указанных в записи деталей со склада; при добавлении/изменении механиков и склада формируются уведомления.

### 2.2. Реализация ключевых функций

**Расписание.** На фронтенде отображаются календарная сетка месяца и блок «День DD.MM.YYYY» с сеткой часов (9–21). Записи загружаются с /api/appointments и группируются по дню и часу; позиция по вертикали внутри часа вычисляется по минутам. Форма записи содержит все перечисленные поля; при сохранении при editingId выполняется PUT, иначе POST. В модальном окне по двойному клику показываются полные данные записи и кнопки «Редактировать» и «Удалить».

**Склад и детали в записи.** В форме записи добавлен блок «Необходимые детали»: выбор позиции из списка запчастей, количество, кнопка «+» для добавления в список. Список сохраняется в поле requiredParts записи. На бэкенде при переходе записи в статус COMPLETED вызывается функция списания по requiredParts (уменьшение quantity в data.json).

**Отчёты.** Запрос к /api/reports с параметром period (ГГГГ-ММ). На сервере по истории за этот период считаются выручка, разбивка по услугам и по механикам (оклад + доплата за каждую выполненную работу). На фронтенде выводятся карточки и таблицы, а также график работ за месяц (линейный график по дням с осью «количество работ» 0–35).

**Уведомления.** При создании/удалении записей, изменении статуса записи, добавлении/изменении механиков (в т.ч. смена активности), добавлении/изменении/удалении позиций на складе в data.json дописываются элементы в массив notifications. В шапке приложения отображается иконка и выпадающий список последних уведомлений с возможностью очистки.

Ограничения ввода (только цифры в телефоне, только буквы в имени, маска госномера, 17 символов VIN и т.д.) реализованы через обработчики keydown и input с использованием общих функций из inputFilters.js.

### 2.3. Визуальное оформление

Использована единая фиолетовая палитра (#7c3aed, оттенки), скруглённые карточки и кнопки, лёгкие тени и анимации (появление модальных окон, перелистывание календаря и дней расписания, hover на кнопках и карточках). График работ выполнен в виде линии с заливкой под ней и подписями по оси Y (0–35 работ) и по оси X (даты). Логотип вынесен в левый верхний угол (sidebar).

## Глава 3. Тестирование и результаты

Проверка проводилась вручную в браузере при локальном запуске (npm run dev в корне проекта): фронтенд на http://localhost:5173, бэкенд на http://localhost:3000.

- Авторизация: вход под менеджером и механиком с корректными данными и капчей проходит; при неверных данных возвращаются сообщения об ошибке.
- Расписание: создание, редактирование и удаление записей работают; позиция записи в сетке по времени соответствует выбранному часу и минутам; при завершении записи количество запчастей на складе уменьшается.
- Склад: добавление и удаление позиций отражается в списке и в отчётах; при выборе деталей в записи и завершении записи остатки обновляются.
- Механики: добавление и смена активности сохраняются; в отчётах считается зарплата (оклад + доплата за количество работ за период).
- Отчёты: при выборе периода отображаются выручка, таблицы по услугам и по механикам, график по дням месяца.
- История: отображаются только завершённые работы; фильтр по госномеру и просмотр деталей по двойному клику работают.
- Уведомления: при действиях в системе в списке появляются новые сообщения; отображение и очистка работают.
- Ограничения полей: в полях имени, телефона, года, госномера, VIN, цены и количества ввод ограничен согласно описанной логике.

В результате получено рабочее одностраничное приложение, готовое к использованию в учебных целях или в малом автосервисе при развёртывании на сервере с Node.js.

## Приложения

**Приложение А.** Инструкция по запуску проекта (фрагмент из README проекта).

- Требования: Node.js v18+, npm.  
- Установка зависимостей: в корне проекта выполнить установку в backend, frontend и в корне (npm install в каждой папке).  
- Запуск: в корне выполнить `npm run dev`; открыть в браузере http://localhost:5173.  
- Тестовые учётные данные: менеджер — manager1 / password123; механик — mech1 / password123; при входе необходимо верно решить капчу (сумма двух чисел).

**Приложение Б.** Структура данных в backend/data.json (основные сущности).

- clients — массив клиентов (id, name, phone, note).  
- services — услуги (id, name, description, duration, price).  
- parts — запчасти (id, article, name, price, quantity).  
- mechanics — механики (id, fullName, position, hireDate, active, baseSalary, bonusPerService).  
- appointments — записи (id, datetime, clientName, phone, carModel, carYear, licensePlate, vin, serviceId, mechanicId, status, comment, requiredParts).  
- history — история (id, appointmentId, datetime, licensePlate, serviceId, mechanicId, status).  
- notifications — уведомления (id, message, timestamp).
